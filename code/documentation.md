# Documentation

This document defines the standards for writing and maintaining documentation across all repositories of the OhMyOpenSource! organization.

Documentation is not a secondary artifact, it is part of the codebase. Undocumented code is incomplete code. A feature that cannot be understood, integrated or maintained without tribal knowledge is a liability.

> A small set of accurate, up-to-date documentation is worth far more than a sprawling collection of docs in various states of disrepair.

---

## Types of Documentation

Documentation lives at different levels of a project and serves different audiences. Understanding which type is needed, and keeping them separate, prevents the common failure of having everything in one place and nothing truly useful anywhere.

### Inline documentation (code comments and docstrings)

The first layer of documentation lives inside the source code itself. Its audience is developers reading or modifying that code.

**Comments** explain *why* something exists, not *what* it does. Code that requires a comment to explain what it does should be simplified first. Comments are appropriate for:

- Complex algorithms or non-obvious logic
- Workarounds for external bugs or constraints, with a reference
- Business rules that are not obvious from the code structure

```python
# Dont: Redundant - the code already says this
# Increment counter
counter += 1

# Do: Explains why, not what
# Rate limit window resets at the top of the hour, not on a rolling basis.
# See: https://api.example.com/docs/rate-limits
if now.minute == 0:
    reset_rate_limit()
```

**Docstrings** document the public interface of a module, class, or function, its purpose, parameters, return values and notable behavior. They are the primary source of API documentation generated by tools like Sphinx, JSDoc or GoDoc, and they appear as inline help in IDEs.

Every public function, class and module must have a docstring. Private or internal functions should have one when their purpose or behavior is not immediately obvious.

```python
# Python - Google style docstring
def paginate(items: list, page: int, page_size: int) -> list:
    """Return a single page of items from a list.

    Args:
        items: The full list of items to paginate.
        page: The 1-based page number to return.
        page_size: The number of items per page.

    Returns:
        A slice of `items` corresponding to the requested page.
        Returns an empty list if `page` is out of range.
    """
```

```typescript
// TypeScript - JSDoc
/**
 * Returns a single page of items from an array.
 *
 * @param items - The full array to paginate.
 * @param page - The 1-based page number to return.
 * @param pageSize - The number of items per page.
 * @returns A slice of `items` for the requested page, or an empty array if out of range.
 */
function paginate<T>(items: T[], page: number, pageSize: number): T[] {}
```

The docstring style to use depends on the language. For most languages, the **Google style** is the reference (see [google.github.io/styleguide](https://google.github.io/styleguide/)).

### README

Every repository must have a `README.md` at its root. The README is the front door of the project — the first thing a new contributor or user sees.

A README must cover, at minimum:

- **What the project is**: one or two sentences describing its purpose
- **How to install or set up**: the exact steps to get a working local environment
- **How to run**: how to start the application or run the tests
- **How to contribute**: a reference to `CONTRIBUTING.md` or brief contribution instructions

A README should be accurate enough that a contributor with no prior context can get the project running by following it alone. If the README is out of date, update it, do not work around it.

```markdown
# project-name

One-sentence description of what this project does and why it exists.

## Getting started

\```bash
git clone https://github.com/ohmyopensource/project-name
cd project-name
cp .env.example .env
npm install
npm run dev
\```

## Running tests

\```bash
npm test
\```

## Contributing

See [CONTRIBUTING.md](CONTRIBUTING.md).
```

### CONTRIBUTING.md

Every repository must have a `CONTRIBUTING.md` that documents the project-specific conventions a contributor needs to know before opening a PR. This includes:

- How to set up the development environment
- Which linter and formatter are in use and how to run them
- Branch naming conventions for this project
- Any deviations from the organization-wide guidelines

GitHub displays `CONTRIBUTING.md` automatically when a contributor opens a PR or issue, making it the most effective place to communicate project-specific expectations.

### CHANGELOG.md

Projects with versioned releases must maintain a `CHANGELOG.md` that documents what changed in each version, following the [Keep a Changelog](https://keepachangelog.com/) format.

```markdown
# Changelog

## [2.3.0] — 2025-10-14

### Added
- Email notifications for new direct messages

### Fixed
- Prevent checkout when cart is empty (#389)

### Changed
- Paginate user list endpoint (returns 20 items per page by default)

## [2.2.1] — 2025-10-02

### Fixed
- Prevent duplicate charge on payment network timeout (#412)
```

The changelog must be updated as part of the release process — not retroactively reconstructed from commit history.

### API Documentation

Public APIs must be documented so that a developer who has never seen the codebase can integrate against them without reading the source code.

API documentation must include, for each endpoint or interface:

- **Purpose**: what it does, expressed as a use case first
- **Request**: method, URL, headers, parameters and request body with types and constraints
- **Response**: status codes, response body structure, error responses
- **Authentication**: how to authenticate
- **Examples**: at least one complete working example per endpoint, including request and response

**Organize by use case, not by endpoint.** The reference listing of all endpoints is necessary, but it should not be the entry point. A developer trying to accomplish something specific should be able to find a worked example first.

```markdown
## Create a user

Creates a new user account. Returns the created user object on success.

### Request

POST /v1/users
Content-Type: application/json
Authorization: Bearer <token>

{
  "email": "user@example.com",
  "name": "Jane Doe",
  "role": "member"
}

### Response

201 Created

{
  "id": "usr_01j8k...",
  "email": "user@example.com",
  "name": "Jane Doe",
  "role": "member",
  "created_at": "2025-10-14T09:00:00Z"
}

### Error responses

| Status | Code | Description |
|---|---|---|
| 400 | `invalid_email` | The email address is not valid |
| 409 | `email_taken` | An account with this email already exists |
| 422 | `missing_field` | A required field is absent |
```

For REST APIs, consider adopting the **OpenAPI Specification** (formerly Swagger) to maintain machine-readable API documentation that can generate interactive docs automatically.

## Documentation as Code

Documentation must be treated the same way as source code: versioned, reviewed and kept in sync with the changes it describes.

This means:

- Documentation lives in the same repository as the code it documents
- Documentation changes are part of the same PR as the code changes they accompany, not separate, not after the fact
- Reviewers must check that documentation has been updated when the behavior it describes changes
- Outdated documentation is a bug, it should be treated and prioritized as such

A PR that changes a public API, a public function signature, a configuration option, or a user-facing behavior **must** include the corresponding documentation update. A PR reviewer should request documentation changes with the same expectation as any other required change.

## Markdown Style

All documentation in this organization is written in Markdown. The reference style guide is [Google's Markdown Style Guide](https://google.github.io/styleguide/docguide/style.html).

Key rules:

- One `#` heading per document (the document title), all other headings start at `##`
- Always declare the language in fenced code blocks: ` ```python `, ` ```bash `, ` ```yaml `
- Use ATX-style headings (`## Heading`) not underline-style
- Wrap lines at 80 characters in source (though rendered output is unaffected)
- Use descriptive link text, never `click here` or bare URLs in prose
- Use reference-style links for long URLs to keep the source readable

```markdown
# Don't
See [here](https://very-long-url.example.com/docs/page?ref=something) for details.

# Do
See the [configuration reference][config-ref] for details.

[config-ref]: https://very-long-url.example.com/docs/page?ref=something
```

## What Not to Document

Over-documentation is as harmful as under-documentation. It creates noise, falls out of date, and discourages contributors from maintaining anything at all.

Do not document:

- What the code already says clearly, a comment that restates the code adds nothing
- Speculative or planned behavior that has not been implemented
- Internal implementation details that are subject to change without notice and are not part of any public interface

Prefer deleting outdated documentation over updating it incorrectly. Incorrect documentation is more dangerous than missing documentation, a reader will follow it and be misled.

## Quick Reference

| Documentation type | Location | Required |
|---|---|---|
| Code comments | Inline in source | When logic is non-obvious |
| Docstrings | Every public function / class / module | Mandatory |
| README | Repository root | Mandatory |
| CONTRIBUTING.md | Repository root | Mandatory |
| CHANGELOG.md | Repository root | For versioned projects |
| API documentation | `docs/` or dedicated doc site | For all public APIs |
